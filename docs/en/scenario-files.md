# Scenario files

## Files you create and files Guidebot creates

The recommended project layout is:

```text
scenarios/
├── account-login.scenario.yaml   # authored source
└── account-login.compiled.yaml   # generated by `guidebot compile`
out/
├── account-login.mp4             # generated by `guidebot render`
└── .guidebot_video/
    └── account-login/             # retained WebM, WAV beds, ffmpeg work
.guidebot/
└── audio/                        # generated narration cache
```

| Path | Created by | Edit manually? | Commit? |
|---|---|---:|---:|
| `*.scenario.yaml` | Author | Yes | Yes |
| `*.render-set.yaml` | Author | Yes | Yes |
| `*.compiled.yaml` | `guidebot compile` | No | Yes |
| `out/*.mp4` | `guidebot render` | No | Usually no |
| `.guidebot/audio/*` | Renderer | No; delete manually when needed | No |
| `<output-dir>/.guidebot_video/<output-stem>/*` | Renderer | No; delete manually when needed | No |

Both generated directories persist after rendering. The audio cache contains MP3
files plus JSON metadata with the narration text; each render workspace may contain
WebM recordings, full-length `bed-<language>.wav` files, composites, and other ffmpeg
intermediates. Treat both as potentially sensitive artifacts.

## Source naming

Use `<name>.scenario.yaml`. The loader also accepts `.scenario.yml`, `.yaml`, and
`.yml`, but the longer suffix tells readers which file is authored source.

The sidecar name is derived automatically:

| Source | Generated sidecar |
|---|---|
| `login.scenario.yaml` | `login.compiled.yaml` |
| `login.scenario.yml` | `login.compiled.yaml` |
| `login.yaml` | `login.compiled.yaml` |
| `login.yml` | `login.compiled.yaml` |

Do not name a source file `login.compiled.yaml`; that suffix is reserved for output.

## Language file strategies

Choose based on what changes:

| Requirement | Files |
|---|---|
| Same page and actions; only narration changes | One source + one sidecar + `audioTracks`/`translations`; see [Multilingual audio](multilingual-audio.md). |
| Page locale, route, labels, or actions change | One complete source + sidecar per locale, grouped by a `*.render-set.yaml`; see [Localized render sets](localized-render-sets.md). |

The stock compiler and renderer now create contexts with the same configured
`locale`. A localized set additionally requires each manifest language key to match
that scenario's `config.locale` and `config.tts.lang`.

## The authored source

The source contains only configuration and ordered intent steps:

```yaml
config:
  title: "Reset a password"
  baseUrl: https://staging.example.com
  viewport: { width: 1280, height: 720 }
  locale: en-US
  tts: { provider: edge, voice: en-US-GuyNeural, lang: en-US }

steps:
  - navigate: /forgot-password
  - enterText:
      into: "the email address field"
      text: "${DEMO_EMAIL}"
    say: "Enter the account email address."
  - teach: "Click the Send reset link button"
```

Comments and formatting remain yours because compilation never modifies this file.
Unknown keys fail validation instead of being silently ignored.

## Slide cards and render-only polish

`config.typing`, `config.sound`, `config.intro`, and a bigger built-in cursor
(`config.cursor.width`/`height`/`click`) are opt-in, render-only cosmetics: turning
them on or off never needs a recompile. A `slide` step is different — it is a step
like any other, so adding, removing, or reordering one changes the step count and
needs `guidebot compile`.

```yaml
config:
  title: "Reset a password"
  viewport: { width: 1280, height: 720 }
  tts: { provider: edge, voice: en-US-GuyNeural, lang: en-US }
  cursor: { width: 46, height: 62 }
  typing: { animate: true, speed: 45 }
  sound: { enabled: true }
  intro: { enabled: true, subtitle: "Password reset walkthrough" }

steps:
  - slide:
      title: "Reset a password"
      subtitle: "Step by step"
    say: "In this video, I'll show you how to reset your password."
  - navigate: /forgot-password
  - enterText:
      into: "the email address field"
      text: "${DEMO_EMAIL}"
    say: "Enter the account email address."
  - teach: "Click the Send reset link button"
  - slide:
      title: "Done"
      hold: 3
```

The opening `slide` narrates over a full-frame title card instead of the blank page,
and the closing `slide` has no `say`, so it simply holds for `hold` seconds after the
last narrated step ends.

## The generated sidecar

After compilation, the adjacent file resembles:

```yaml
# GENERATED by `guidebot compile` (and `render`, for optional branches) — do not edit by hand.
compiler_version: 2
source: reset-password.scenario.yaml
actions:
  - null
  - action: type
    target:
      strategy: role
      role: textbox
      name: Email
      exact: true
    identity:
      tag: input
      testid: email
      ancestry_digest: "..."
      identity_version: 1
    expect: none
    fingerprint:
      command_kind: enterText
      compiled_from: the email address field
      expect: none
      compiler_version: 2
      config_hash: "..."
  - action: click
    # remaining generated target and identity data...
```

`actions` is aligned by index with `steps`. Non-target steps such as either form of
`navigate`, `say`, and numeric `wait` get `null`. `chrome` and alternate-audio
configuration are not copied into the sidecar. Compiler v2 may also store
`input_text` for a validated literal `teach → type` action and `opens_popup: true`
for a click that opened the supported pop-up.

The browser-free fast check verifies source name, compiler version, action count,
null-slot alignment, command kind, target instruction, wait state, readiness, and
the target config hash. It cannot inspect the live application or fingerprint a
direct navigation URL. Use `--force` after route, `baseUrl`, account-state, or page
drift.

The file is written atomically after each completed step. If step 8 fails, valid
progress from steps 1–7 remains available for a later run. With an older sidecar,
unprocessed later slots can remain stale, and the file has no completion marker. Do
not render after a failed compile; finish a successful compile first, using
`--force` when in doubt.

## Review and version control

Review a sidecar as generated code:

- confirm the `source` name and number of action slots;
- look for unexpectedly broad targets, `nth` use, or surprising scopes;
- check whether a changed target is expected after a UI change;
- never insert secrets or hand-patch identities;
- commit the source and sidecar in the same change.

Rendering without the sidecar fails. Before TTS or browser use, render checks source
name, compiler version, action count and kinds, fingerprints, wait states, and target
configuration. It then validates live identity for click, hover, and type actions.
Conditional waits skip that live identity check, but their v2 fingerprint and state
are still preflighted. Route-only and application-only drift still require a
successful `compile --force`. Existing narration, alternate audio, `cursor`, and
`chrome` changes are render-only.

## Environment values and secrets

Write replaceable values as `${NAME}` only in these locations:

```yaml
steps:
  - navigate: "${APP_URL}/login"
  - navigate: { url: "${APP_URL}/reports", type: false }
  - enterText: { into: "the email field", text: "${DEMO_EMAIL}" }
```

Guidebot expands environment values in memory and never writes the expanded value to
the sidecar. Both string `navigate` and object-form `navigate.url` are supported. It
does **not** expand variables in:

- `config.baseUrl`;
- `say` or `teach`;
- `enterText.into`;
- `wait.until`;
- other config fields.

Missing values cause a hard error, including during `validate`. A `.env` file is not
loaded automatically. Use `export`, a shell-local assignment, or your secret runner:

```bash
DEMO_EMAIL=user@example.com \
  uv run guidebot compile scenarios/login.scenario.yaml
```

Write `$${NAME}` when the literal output must be `${NAME}`.

!!! danger "A secret may still leave the process"

    `enterText.text` is not sent directly to the reasoner or written to the sidecar,
    but this is not a general redaction guarantee. A value reflected later into page
    text or an accessible name can enter a later candidate snapshot and generated
    target. It can also be visible in the recording, browser/application logs, and
    retained audio or render artifacts. Use synthetic credentials, password-type
    inputs, and a non-production environment.

!!! warning "An empty scenario cannot be rendered"

    A normal first compile creates an all-`null` sidecar for a nonempty scenario that
    contains only `say`, `navigate`, or numeric `wait`. With `steps: []`, the compiler
    writes no sidecar even with `--force`, so stock `render` is unsupported.

## When render is enough

| Change | Required action |
|---|---|
| Existing `say` text | Render; missing narration is regenerated. |
| `enterText.text` or its environment value | Usually render; the field target is unchanged. |
| `tts.voice`, `provider`, `model`, or `speed` | Render; the audio cache key changes. |
| Alternate `audioTracks`, `translations`, `title`, or `trackLanguage` | Render only; MP4-only metadata does not invalidate TTS cache. |
| Any `cursor` field, including `cursor.click` | Render only. |
| Any `chrome` field or `navigate.type` only | Render only; no target or sidecar rebuild. |
| Any `typing`, `sound`, or `intro` field | Render only. |
| `config.selects.settleMs`, `maxVisibleOptions`, or `openHoldMs` | Render only; cosmetic shim tuning. |
| Target instruction in `teach`/`click`/`hover`/`into`/`until` | Compile. |
| `viewport`, `locale`, or `tts.lang` | Compile; the target config hash changes. |
| `config.selects.mode` (global) switched between `shim`/`native`, or a step's `select.mode` | Compile; the config hash or the step's own fingerprint changes. |
| Application DOM/content drift | `compile --force`. |
| Navigation URL or `baseUrl` | `compile --force`; neither describes a target fingerprint. |
| Step kind/order/addition/removal, including adding/removing/reordering a `slide` step | Compile; use `--force` if route/application state also changed. |
| Guidebot upgrade | Compile; an older compiler version is rejected automatically. |
| Render identity error | `compile --force`. |

The chrome overlay changes only the recorded WebM/MP4 in the existing render work
directory. It creates no new file type. Because it reserves vertical space inside
the viewport, still review the rendered layout even though recompilation is not
required.

For multilingual output, durable full-length beds are named
`bed-<trackLanguage>.wav` inside the output's private workspace. Successful rerenders
replace the complete bed set and remove stale language beds; unrelated recordings
remain until manually removed.

## Lifecycle checklist

1. Create or edit `flow.scenario.yaml`.
2. Export required environment variables.
3. Run `guidebot validate`.
4. Run `guidebot compile` against a resettable environment.
5. Review `flow.compiled.yaml` and commit both YAML files.
6. Run `guidebot render` to create the MP4.
7. After site drift, run compile with `--force`, review the sidecar diff, and render
   again.
